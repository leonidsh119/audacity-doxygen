36 /* Design:
37 
38 This operator implements sequences of multichannel signals.
39 A single data structure manages an array of susps that
40 initially are used to fetch blocks from the first multichannel
41 signal. When the LAST logical stop is reached, a closure
42 is evaluated to yield a new multichannel signal. The component
43 sounds of this are stored into the susps which are then
44 converted into add suspensions. The other managing structures
45 are then freed.
46 
47 The main constraint here is that the conversion to add susps
48 must take place at the same time across all channels, so before
49 the conversion, a fetch from the susp can only be made if it is
50 known that the samples returned happen BEFORE the conversion will
51 take place. Since the conversion takes place at the maximum of
52 the logical stop times of all channels, we have to advance all
53 channels synchronously. We keep track of a greatest lower bound,
54 refered to as the horizon, on the maximum logical stop time. It
55 is safe to fetch blocks up to the horizon, but not beyond.
56 
57 This synchronous fetching is done by a single routine and an
58 auxilliarly structure that manages the whole multichannel array
59 of susps. The basic idea is that a fetch from a
60 suspension gets forwarded to the managing structure, which
61 uses its array of susps to fetch from ALL suspensions up to
62 the requested time or until the logical stop, whichever comes
63 first. These "synchronous" fetches are not made by calling the
64 fetch routines on the suspensions to avoid infinite recursion.
65 
66 At any time, there will be some set of channels whose logical
67 stop time is unknown. The "s1" fields (s1, s1_ptr, s1_bptr) of
68 these suspensions are used to look ahead by geting a block from
69 s1. If no logical stop is indicated, then we can append the block
70 to the snd_list and update the horizon, allowing fetches from other
71 susps. In other words, the s1_bptr of each susp provides a one
72 buffer lookahead by which we can obtain advance knowledge of the
73 maximum logical stop time.
74 
75 The algorithm is as follows:
76 
77 1. When fetch is called on a suspension, compute when any
78 prefetched samples will end (if there are none, then fetch
79 a block from s1 and compute the time at which the block ends).
80 This becomes the target time for other fetches.
81 
82 2. Call multiseq_advance(), passing the target time and the
83 manager structure (which has pointers to all other channels).
84 (Note: every susp has a pointer to the manager).
85 The function of multiseq_advance() is to push the horizon for
86 the logical stop forward. This is done by
87 iterating over the array of susps until the target is reached.
88 Actually, the array contains pointers to the snd_list_node that
89 points to each susp and where the next block will be linked.
90 The goal of this loop is to satisfy the original fetch, which
91 means we have to push low_water to greater than or equal to the
92 target. (Low water is the minimum time of the next sample to
93 be returned by any multiseq susp.) This goal will be met unless
94 we reach the last logical stop time, in which case we evaluate
95 the closure for the next multichannel sound, convert
96 everything to add's and let the additions take care of returning
97 blocks.
98 
99 3. The Iteration Loop:
100 low_water is the lowest sample count of the next sample
101 horizon is the greatest lower bound on the maximum logical stop time
102 Iterate over susps until low_water >= target
103 (Note: whenever data is fetched for a sound whose logical stop time
104 is unknown, update the horizon. If a logical stop time becomes known,
105 then test if the final maximum logical stop time is known (by keeping
106 a count of how many are still unknown), and if the count goes to zero,
107 evaluate the continuation and convert to multiple adds.)
108 (Another Note: we may reach the logical stop time and convert to
109 multiple adds before the loop terminates, in which case we return
110 without finishing the loop. Take care that the caller does the right
111 thing to produce a sample block in this case.)
112 
113 3a. If a block hasn't been prefetched, do it.
114 
115 3b. While the susp has prefetched a block that ends at or before horizon,
116 put the block on the snd_list and prefetch another block.
117 
118 3c. If the susp hasn't a known logical stop time, set new_horizon to
119 the end time of the last sample prefetched in 3b.
120 
121 3d. If new_horizon == horizon, signal an error, no progress was made.
122 
123 3d. Set horizon to new_horizon and repeat the loop.
124 
125 NOTE ON A BUG FIX (1 Jul 95): old code assumed that when a logical stop
126 was detected it was at the beginning of the next block, but if logical
127 stop is explicit, then it may be way in the future. We could convert
128 to adds at this point, but that would force early evaluation of the
129 closure, which we'd like to delay (be lazy when possible). Therefore,
130 we want to ignore knowledge of a logical stop time until the logical
131 stop time falls within the currently known block of samples. By "currently
132 known", I mean somewhere in the block referenced by ->s1_ptr and ->s1_cnt.
133 
134 */

293  if (my_low_water < new_low_water) {

295  }

297  }
298  ms->low_water = new_low_water;
299  if (new_horizon <= ms->horizon) {
300  stdputstr("no progress in multiseq_advance\n");
301  EXIT(1);
302  } else {
303  ms->horizon = new_horizon;
304  }
305  }

414  * look alike to multiseq_advance. Of course, multiseq_advance will
415  * redo the work of allocating a snd_list.

418  * fetch will return pointers to the zero_block, but this will

478 D nyquist_printf("block_end_time of %p: %g\n", susp, block_end_time);
