Normal usage should be\+:


\begin{DoxyItemize}
\item Plugins include \hyperlink{libvamp_2vamp-sdk_2plugin_8h}{vamp-\/sdk/\+Plugin.\+h} or \hyperlink{vamp-sdk_2_plugin_base_8h}{vamp-\/sdk/\+Plugin\+Base.\+h}. These files include this header, which specifies an appropriate namespace for the plugin classes to avoid any risk of conflict with non-\/plugin class implementations in the host on load.
\item Hosts include \hyperlink{libvamp_2vamp-hostsdk_2plugin_8h}{vamp-\/hostsdk/\+Plugin.\+h}, \hyperlink{vamp-hostsdk_2_plugin_base_8h}{vamp-\/hostsdk/\+Plugin\+Base.\+h}, vamp-\/hostsdk/\+Plugin\+Host\+Adapter, \hyperlink{_plugin_loader_8h}{vamp-\/hostsdk/\+Plugin\+Loader.\+h} etc. These files include \hyperlink{hostguard_8h}{vamp-\/hostsdk/hostguard.\+h}, which makes a note that we are in a host. A file such as \hyperlink{libvamp_2vamp-hostsdk_2plugin_8h}{vamp-\/hostsdk/\+Plugin.\+h} then simply includes \hyperlink{libvamp_2vamp-sdk_2plugin_8h}{vamp-\/sdk/\+Plugin.\+h}, and this guard header takes notice of the fact that it has been included from a host and leaves the plugin namespace unset.
\end{DoxyItemize}

Problems will occur when a host includes files directly from the vamp-\/sdk directory. There are two reasons this might happen\+: mistake, perhaps owing to ignorance of the fact that this isn\textquotesingle{}t allowed (particularly since it was the normal mechanism in v1 of the S\+DK); and a wish to incorporate plugin code directly into the host rather than having to load it.

What if the host does include a vamp-\/sdk header by mistake? We can catch it if it\textquotesingle{}s included before something from vamp-\/hostsdk. If it\textquotesingle{}s included after something from vamp-\/hostsdk, it will work OK anyway. The remaining problem case is where nothing from vamp-\/hostsdk is included in the same file. We can\textquotesingle{}t catch that.
