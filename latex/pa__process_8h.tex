\subsection{Detailed Description}
Buffer Processor prototypes. A Buffer Processor performs buffer length adaption, coordinates sample format conversion, and interleaves/deinterleaves channels. 

\paragraph*{Overview}

The \char`\"{}\+Buffer Processor\char`\"{} (\hyperlink{struct_pa_util_buffer_processor}{Pa\+Util\+Buffer\+Processor}) manages conversion of audio data from host buffers to user buffers and back again. Where required, the buffer processor takes care of converting between host and user sample formats, interleaving and deinterleaving multichannel buffers, and adapting between host and user buffers with different lengths. The buffer processor may be used with full and half duplex streams, for both callback streams and blocking read/write streams.

One of the important capabilities provided by the buffer processor is the ability to adapt between user and host buffer sizes of different lengths with minimum latency. Although this task is relatively easy to perform when the host buffer size is an integer multiple of the user buffer size, the problem is more complicated when this is not the case -\/ especially for full-\/duplex callback streams. Where necessary the adaption is implemented by internally buffering some input and/or output data. The buffer adation algorithm used by the buffer processor was originally implemented by Stephan Letz for the A\+S\+IO version of Port\+Audio, and is described in his Callback\+\_\+adaption\+\_\+.\+pdf which is included in the distribution.

The buffer processor performs sample conversion using the functions provided by pa\+\_\+converters.\+c.

The following sections provide an overview of how to use the buffer processor. Interested readers are advised to consult the host A\+PI implementations for examples of buffer processor usage.

\subparagraph*{Initialization, resetting and termination}

When a stream is opened, the buffer processor should be initialized using Pa\+Util\+\_\+\+Initialize\+Buffer\+Processor. This function initializes internal state and allocates temporary buffers as neccesary according to the supplied configuration parameters. Some of the parameters correspond to those requested by the user in their call to \hyperlink{portaudio_8h_a443ad16338191af364e3be988014cbbe}{Pa\+\_\+\+Open\+Stream()}, others reflect the requirements of the host A\+PI implementation -\/ they indicate host buffer sizes, formats, and the type of buffering which the Host A\+PI uses. The buffer processor should be initialized for callback streams and blocking read/write streams.

Call Pa\+Util\+\_\+\+Reset\+Buffer\+Processor to clear any sample data which is present in the buffer processor before starting to use it (for example when Pa\+\_\+\+Start\+Stream is called).

When the buffer processor is no longer used call Pa\+Util\+\_\+\+Terminate\+Buffer\+Processor.

\subparagraph*{Using the buffer processor for a callback stream}

The buffer processor\textquotesingle{}s role in a callback stream is to take host input buffers process them with the stream callback, and fill host output buffers. For a full duplex stream, the buffer processor handles input and output simultaneously due to the requirements of the minimum-\/latency buffer adation algorithm.

When a host buffer becomes available, the implementation should call the buffer processor to process the buffer. The buffer processor calls the stream callback to consume and/or produce audio data as necessary. The buffer processor will convert sample formats, interleave/deinterleave channels, and slice or chunk the data to the appropriate buffer lengths according to the requirements of the stream callback and the host A\+PI.

To process a host buffer (or a pair of host buffers for a full-\/duplex stream) use the following calling sequence\+:


\begin{DoxyEnumerate}
\item Call Pa\+Util\+\_\+\+Begin\+Buffer\+Processing
\item For a stream which takes input\+:
\begin{DoxyItemize}
\item Call Pa\+Util\+\_\+\+Set\+Input\+Frame\+Count with the number of frames in the host input buffer.
\item Call one of the following functions one or more times to tell the buffer processor about the host input \hyperlink{xmlparse_8c_aaadc06b5dac8070de2c8677210967bdb}{buffer(s)}\+: Pa\+Util\+\_\+\+Set\+Input\+Channel, Pa\+Util\+\_\+\+Set\+Interleaved\+Input\+Channels, Pa\+Util\+\_\+\+Set\+Non\+Interleaved\+Input\+Channel. Which function you call will depend on whether the host \hyperlink{xmlparse_8c_aaadc06b5dac8070de2c8677210967bdb}{buffer(s)} are interleaved or not.
\item If the available host data is split accross two buffers (for example a data range at the end of a circular buffer and another range at the beginning of the circular buffer), also call Pa\+Util\+\_\+\+Set2nd\+Input\+Frame\+Count, Pa\+Util\+\_\+\+Set2nd\+Input\+Channel, Pa\+Util\+\_\+\+Set2nd\+Interleaved\+Input\+Channels, Pa\+Util\+\_\+\+Set2nd\+Non\+Interleaved\+Input\+Channel as necessary to tell the buffer processor about the second buffer.
\end{DoxyItemize}
\item For a stream which generates output\+:
\begin{DoxyItemize}
\item Call Pa\+Util\+\_\+\+Set\+Output\+Frame\+Count with the number of frames in the host output buffer.

\subparagraph*{Using the buffer processor for a blocking read/write stream}

Blocking read/write streams use the buffer processor to convert and copy user output data to a host buffer, and to convert and copy host input data to the user\textquotesingle{}s buffer. The buffer processor does not perform any buffer adaption. When using the buffer processor in a blocking read/write stream the input and output conversion are performed separately by the Pa\+Util\+\_\+\+Copy\+Input and Pa\+Util\+\_\+\+Copy\+Output functions.

To copy data from a host input buffer to the \hyperlink{xmlparse_8c_aaadc06b5dac8070de2c8677210967bdb}{buffer(s)} which the user supplies to Pa\+\_\+\+Read\+Stream, use the following calling sequence.


\begin{DoxyItemize}
\item Repeat the following three steps until the user \hyperlink{xmlparse_8c_aaadc06b5dac8070de2c8677210967bdb}{buffer(s)} have been filled with samples from the host input buffers\+:
\begin{DoxyEnumerate}
\item Call Pa\+Util\+\_\+\+Set\+Input\+Frame\+Count with the number of frames in the host input buffer.
\item Call one of the following functions one or more times to tell the buffer processor about the host input \hyperlink{xmlparse_8c_aaadc06b5dac8070de2c8677210967bdb}{buffer(s)}\+: Pa\+Util\+\_\+\+Set\+Input\+Channel, Pa\+Util\+\_\+\+Set\+Interleaved\+Input\+Channels, Pa\+Util\+\_\+\+Set\+Non\+Interleaved\+Input\+Channel. Which function you call will depend on whether the host \hyperlink{xmlparse_8c_aaadc06b5dac8070de2c8677210967bdb}{buffer(s)} are interleaved or not.
\item Call Pa\+Util\+\_\+\+Copy\+Input with the user buffer pointer (or a copy of the array of buffer pointers for a non-\/interleaved stream) passed to Pa\+\_\+\+Read\+Stream, along with the number of frames in the user \hyperlink{xmlparse_8c_aaadc06b5dac8070de2c8677210967bdb}{buffer(s)}. Be careful to pass a {\itshape copy} of the user buffer pointers to Pa\+Util\+\_\+\+Copy\+Input because Pa\+Util\+\_\+\+Copy\+Input advances the pointers to the start of the next region to copy.
\end{DoxyEnumerate}
\item Pa\+Util\+\_\+\+Copy\+Input will not copy more data than is available in the host \hyperlink{xmlparse_8c_aaadc06b5dac8070de2c8677210967bdb}{buffer(s)}, so the above steps need to be repeated until the user \hyperlink{xmlparse_8c_aaadc06b5dac8070de2c8677210967bdb}{buffer(s)} are full.
\end{DoxyItemize}

To copy data to the host output buffer from the user buffers(s) supplied to Pa\+\_\+\+Write\+Stream use the following calling sequence.


\begin{DoxyItemize}
\item Repeat the following three steps until all frames from the user \hyperlink{xmlparse_8c_aaadc06b5dac8070de2c8677210967bdb}{buffer(s)} have been copied to the host A\+PI\+:
\begin{DoxyEnumerate}
\item Call Pa\+Util\+\_\+\+Set\+Output\+Frame\+Count with the number of frames in the host output buffer.
\item Call one of the following functions one or more times to tell the buffer processor about the host output \hyperlink{xmlparse_8c_aaadc06b5dac8070de2c8677210967bdb}{buffer(s)}\+: Pa\+Util\+\_\+\+Set\+Output\+Channel, Pa\+Util\+\_\+\+Set\+Interleaved\+Output\+Channels, Pa\+Util\+\_\+\+Set\+Non\+Interleaved\+Output\+Channel. Which function you call will depend on whether the host \hyperlink{xmlparse_8c_aaadc06b5dac8070de2c8677210967bdb}{buffer(s)} are interleaved or not.
\item Call Pa\+Util\+\_\+\+Copy\+Output with the user buffer pointer (or a copy of the array of buffer pointers for a non-\/interleaved stream) passed to Pa\+\_\+\+Write\+Stream, along with the number of frames in the user \hyperlink{xmlparse_8c_aaadc06b5dac8070de2c8677210967bdb}{buffer(s)}. Be careful to pass a {\itshape copy} of the user buffer pointers to Pa\+Util\+\_\+\+Copy\+Output because Pa\+Util\+\_\+\+Copy\+Output advances the pointers to the start of the next region to copy.
\end{DoxyEnumerate}
\item Pa\+Util\+\_\+\+Copy\+Output will not copy more data than fits in the host \hyperlink{xmlparse_8c_aaadc06b5dac8070de2c8677210967bdb}{buffer(s)}, so the above steps need to be repeated until all user data is copied. 
\end{DoxyItemize}
